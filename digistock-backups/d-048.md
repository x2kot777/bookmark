# @sawa20200424 (すがさわ｜事業家) - X 投稿

Author: すがさわ｜事業家 (@sawa20200424)

Tweet text:
━━━━━━━━━━━━
フォーム営業自動化 指示書
━━━━━━━━━━━━
全体構成
4つの AI エージェントを直列に動かす。
最後に人間がコマンド1つ叩いて完了。
①リサーチ → ②リッチ化 → ③試運転(dry-run) → ④送信

①リサーチエージェント
━━━━━━━━━━━
やること：公開情報から「会社名 + 公式URL」を抽出してリスト化
使うソース（URLがもとから書かれている情報源のみ）：

PRtimes のプレスリリース本文（「URL：https://〜」パターンを正規表現で抽出）

経産省 IT導入支援事業者一覧PDF（pdfplumber で抽出）
SaaSパートナー一覧（HubSpot / kintone / Salesforce / Shopify）
業界団体メンバー一覧
重要：検索エンジンスクレイピング（Yahoo / Google / DuckDuckGo）は 2026年現在 anti-bot対策で全滅。やめとけ。

②リッチ化エージェント
━━━━━━━━━━━
やること：会社の公式URLから問い合わせフォームURLを発見
ロジック：

公式URL に対して /contact/ /inquiry/ /お問い合わせ/ /contact-us/ を順番に GET

HTML 内に <form> + <textarea> + <input type=text|email> がある → フォームページ確定
requests + BeautifulSoup（Python）で十分
事故防止チェック（必須）：
会社名が公式URLのHTMLに含まれているか確認（含まれない＝別会社の可能性）
公式URLと問い合わせURLの root_domain が一致しているか確認

同じドメインに複数会社が紐づいたら全部隔離

③試運転エージェント（dry-run）
━━━━━━━━━━━━━━━
やること：Playwright でフォームを自動入力して 送信ボタン手前で停止
ツール：playwright（Node.js）+ CDP接続でChrome操作
adapter（フォーム種別ごとの入力ロジック）：

Contact Form 7（WordPress プラグイン）

WPForms

MW WP Form
generic_form（汎用フォールバック）
判定：必須項目すべて埋まる + 入力検査NGなし → OK
audit（送信前 safe判定）：
toC会社（個人向け）除外

公的機関・大企業除外

第三者プラットフォーム（http://form-mailer.jp等）除外
用途違いURL（/recruit / /support）除外

④送信実行エージェント
━━━━━━━━━━━
やること：dry-run OK だった会社にだけ本番送信
コマンド例：
node submit_batch.mjs --dry-run=false --input=safe.csv

注意点：

dry-run と LIVE で挙動が違うフォームがある
確認画面で動的に必須項目が増えるフォームでLIVE失敗多発

「業種」「お問い合わせ項目」「ふりがな（ひらがな指定）」は要対応

落とし穴（事故防止）
━━━━━━━━━━
PRtimes本文のURL = 公式URLとは限らないリリース本文に出てくる URL は提携先・イベント会場・販売ページの可能性
必ず「公式URLのHTML内に会社名が含まれるか」確認すべし

既存資産の確認を最初にやる新規実装ゼロから書く前に、既存CSVの「本文」列を必ず読む
送信文面を勝手に省略すると78社に短文版が届く事故が起きる（実体験）

送信失敗の再送はROI低いフォーム側の動的バリデーションで失敗したものは、汎用adapter改修では救えない
1社ずつ個別対応するより、新規会社採集に時間を使うべき

スケール感
━━━━━
公開情報からの採集：1日5万件レベル可能
フォーム発見率：100-300件（残り99%は contact URL なし or toC）
送信成功率：30-40%
アポ率：1%

結論：
━━━
仕組みを作るより、送る文章とオファーの方が100倍重要。
仕組みは2-3日で作れるが、オファー設計に2-3ヶ月かける価値あり。
